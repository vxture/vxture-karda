# Karda 加工管线设计(Processing Pipeline)(110-processing)

> 版本:v0.1
> 状态:Draft — 待拍板项 4 项(见 §11)
> 上游文档:10-product-definition v0.4、100-kb-model(对象模型/模板/状态机)、120-retrieval-tools(检索侧约束)、200-arda-channel(Arda 通道契约)、product_110/210
> 定位:Karda 文档族第三份纵深设计,定义摄取到索引的离线加工管线——任务架构、多阶段解析、模板化分块、向量化与原子提交、增量更新、受控重建、失败与重试
> 设计方法:按行业最佳实践设计——离线索引与在线检索分离、异步非阻塞摄取、阶段产物留存、仅处理 delta、文档级原子替换、build-then-swap 重建

---

## 1. 设计原则(行业收敛)

| # | 原则 | 依据 |
|---|---|---|
| 1 | **离线/在线分离**:加工管线与检索服务为两条独立工作流,加工变更不触碰在线路径 | 生产 RAG 通行范式:offline indexing 与 online retrieval 分离才有可调试性 |
| 2 | **异步非阻塞**:摄取全部队列化,同步阻塞式管线无法承受万级文档 | 行业一致结论 |
| 3 | **阶段产物留存**:原始件与解析中间表示持久化——重分块不重解析,重索引不重下载 | raw preservation 与可追溯实践(Databricks 等) |
| 4 | **仅处理 delta**:增量为常态,以 content_hash 为变更判据;全量重处理不可扩展 | 行业一致结论(承接 200-arda-channel §6) |
| 5 | **文档级原子替换**:一个文档的新分块集合整体替换旧集合,检索永不见半更新态 | stable IDs + 确定性更新/删除的最小契约 |
| 6 | **重建 = build-then-swap**:影子索引后台构建、完成原子切换,期间检索用旧索引 | 索引蓝绿切换通行做法 |

---

## 2. 管线总览

```
摄取入口(三合一)                          离线加工管线(队列化,按文档为任务单元)
  上传(Console/API)──┐
  karda.write_*(OBO)─┼─→ 落 Document(processing)─→ [fetch]→[parse]→[chunk]→[embed]→[commit]
  Arda 通道(200-arda-channel)┘        │                      │每阶段产物持久化,失败可从断点续
                               │                      ▼
                               │              indexed(原子替换生效)/ failed(驻留)
                               └── 状态与进度回写(Console 失败件视图、召回测试可用性)
在线检索(120-retrieval-tools)只读消费已提交索引,与本管线零耦合
```

`processing` 子状态即五阶段:`fetching → parsing → chunking → embedding → committing`(对 100-kb-model §5.1 的细化);任一阶段失败按 §8 分类处置。

---

## 3. 任务架构

- **任务单元 = 文档**:每文档一个加工任务,携带 (kb_id, doc_id, template, trigger);任务幂等键 = (doc_id, content_hash, config_fingerprint)——同键任务去重;
- **队列分级**:`interactive`(用户上传/Entry 提交,秒级期望)> `sync`(Arda 增量)> `bulk`(backfill / 重建 / 实例化)。三级队列独立并发池,bulk 永不饿死 interactive;
- **并发与公平性**:org 级并发上限(多租户公平)+ KB 级串行窗口(同库文档避免索引写争抢);背压:队列深度超阈值时 Arda 通道自然背压(notify-then-pull 拉取放缓,200-arda-channel §5),上传入口返回排队位次;
- **凭证姿态**:后台加工调用 Atlas(embedding/解析模型)以 **service 模式**铸币,workspace = **库的归属 WS**(U 级 = home_ws,T 级 = 建设 WS)——加工成本记到资产归属方,而非触发方;OBO 写入仅用于入口鉴权与落件,落件即转后台;
- **配额前置检查**:任务入队前查存储配额与加工配额(C3),超限拒绝入队(`409 quota_exhausted`),不产生半成品。

## 4. 多阶段解析(parse 阶段)

### 4.1 双路径设计

| 路径 | 适用 | 阶段 | 成本 |
|---|---|---|---|
| **fast path** | 原生文本类:md/txt/html/docx(带文本层 pdf) | 类型嗅探 → 格式归一 → 结构树构建(标题层级/列表/表格标记) | CPU,毫秒~秒级 |
| **deep path** | 扫描件/图片/复杂版面 pdf/ppt | 类型嗅探 → 版面分析 → 块类型识别(标题/正文/表格/公式/图)→ 内容提取(OCR / 表格结构还原 / 公式转 LaTeX)→ 结构树构建 → 语义修复 | 视觉模型,秒~分钟级 |

对标 RAGFlow deepdoc(视觉处理 + 解析器两部分:OCR、版面识别、TSR),但有一处结构性差异:

**解析用视觉/小模型(版面分析、OCR、表格结构、公式识别)一律注册为 Atlas 专用模型,经 Atlas 调用**——铁律"Atlas 之外不存在第二个模型宿主"无例外;Karda 解析器只含规则与编排逻辑,不内嵌模型运行时(区别于 RAGFlow 的内嵌式)。工程影响:解析类调用高频、小请求、延迟敏感,需要 Atlas 支持**批量接口与部署亲和**(解析模型端点与 Karda worker 同域部署)——登记为对 Atlas 的需求项(§12)。

### 4.2 解析产物:结构化中间表示(IR)

解析输出统一为**元素树**(element tree:章节层级 + 元素类型 + 内容 + 位置锚点),持久化存储:

- 分块、重分块只消费 IR,不回读原始件;模板调参重加工 = 从 chunk 阶段重跑,省去 deep path 成本;
- 位置锚点(页码/坐标/段落序)沉淀为 Chunk 的 `locator`(120-retrieval-tools §5 引用溯源的数据源);
- IR 版本随解析器版本标记,解析器升级触发的重建可按版本圈定范围。

## 5. 模板化分块(chunk 阶段)

ProcessingTemplate(100-kb-model 预置六种)在本阶段的行为矩阵:

| 模板 | 解析路径偏好 | 分块策略 | 上下文增强 |
|---|---|---|---|
| general(默认) | 自动 | **结构感知 + 语义边界**:按 IR 元素边界切分,超长元素内按语义断点二分;不做纯定长切分(行业结论:定长是教程默认、生产错解;语义分块检索精度显著更优) | 标题路径前缀(contextual chunking:章节层级拼入块首) |
| qa | 自动 | 问答对拆分:一问一答一块;文件批量导入按模板规则识别 QA 边界 | 问题加权入检索文本 |
| table | 偏 deep(TSR) | 行式分块:表头 + 行内容为一块 | 表头复写每块 |
| manual | 自动 | 层级保留:父子分块(父 = 章节,子 = 段落;召回子块携带父上下文) | 章节路径 + 父块引用 |
| paper | 偏 deep | 章节感知 + 公式/图表独立成块并关联所在段 | 摘要/章节前缀 |
| legal | fast 为主 | 条款式:条/款/项为天然块边界,保留条号 | 法规名 + 条号前缀(服务 P 级法规包引用精度) |

参数面(org 可调,100-kb-model §7):目标块长(默认 512 token)、最大块长(1024)、重叠(默认 0——结构感知下重叠价值低,定长退化场景才需要)、自定义分隔符(qa/table 模板)。

Entry 不走本阶段:按 ContentTemplate 字段检索角色直接拼装检索文本(100-kb-model §6)。

## 6. 向量化与原子提交(embed / commit 阶段)

- embedding 批量调用 Atlas(库级锁定模型版本);`429` 退避重试,配额尽则任务挂起(不落 failed,配额恢复自动续跑,状态标 `suspended_quota`);
- 全文索引与向量索引同批构建;
- **commit = 文档级原子替换**:新 chunk 集合以新版本号写入,双版本瞬时并存,提交指针原子切换后异步清理旧版本——检索(120-retrieval-tools)只读已提交版本,永不见半更新;
- 提交后回写 `indexed`,触发库统计与召回测试可用性刷新。

## 7. 增量更新与受控重建

### 7.1 文档级增量(常态)

- 变更判据 = content_hash(上传/API 由 Karda 计算,Arda 通道由信封携带);hash 未变直接跳过(伪变更免疫);
- 变更 = 全文档重加工 + 原子替换(不做块级 diff——工程复杂度高、收益低,行业主流同此);删除 = 索引级联清除(含墓碑事件,200-arda-channel §6)。

### 7.2 库级受控重建(build-then-swap)

触发源:embedding 模型版本变更 / 加工模板或参数变更 / 解析器大版本升级 / silo 迁移。流程:

1. 属主/管理员在 Console 发起(或平台强制,如模型下线),声明变更项;
2. 影子索引后台构建(bulk 队列,复用 IR 能跳过 parse 的尽量跳过);
3. 构建期间在线检索持续使用旧索引,库配置面标注"重建中 + 进度";
4. 完成后**原子切换**,旧索引保留一个回退窗口(默认 24h)后清除;
5. 失败则影子索引废弃,旧索引不受任何影响。

重建属重成本操作:入 bulk 队列受 org 并发上限约束;平台强制类重建(模型下线)由平台排程分批,避免全租户同时重建的资源洪峰。

## 8. 失败分类与重试

| 类别 | 例 | 处置 |
|---|---|---|
| **瞬态**(transient) | fetch_ref 过期、Atlas 429/超时、worker 中断 | 自动重试(指数退避,上限 5 次),从断点阶段续跑;超限转 failed |
| **永久**(permanent) | 格式不支持、文件损坏、解析器确定性失败、超大小限制 | 直接 failed 驻留,不空耗重试 |
| **配额** | 存储/加工配额尽 | `suspended_quota` 挂起态,配额恢复自动续(区别于 failed,不需人工) |

- failed 为显式驻留态(100-kb-model §5.1):Console 失败件视图展示错误分类与详情,属主可修正后手动重试(重试 = 新任务,幂等键含重试代数);
- **毒丸隔离**:单文档失败不阻塞同库/同 Binding 其他文档(承接 200-arda-channel §5);同库失败率超阈值(如 30%)触发库级告警而非继续空转——大概率是模板选错,提示属主调整。

## 9. 包实例化工程(承接 product-definition §4-7)

实例化 = bulk 任务:平台包内容快照复制入 org 存储 → **复用平台侧 IR**(解析产物随包分发,免重解析)→ 按 org 的 embedding 配置重建向量索引(org 模型配置可能异于平台)→ 原子提交为 T 级库(origin_ref 写入)。成本主项仅 embedding,实例化时长可预估并在 Console 展示;并排重同步 = 同流程新实例。

---

## 10. 可观测性(v1 指标基线)

- 管线:各阶段 P50/P95 时长、队列深度(三级)、吞吐(docs/min)、失败率(按分类)、suspended 存量;
- 新鲜度:**ingestion freshness**(源变更 detected_at → indexed 的端到端延迟,Arda 通道 SLO 的度量基础);
- 成本:每库加工用量(解析模型调用、embedding token——均出自 Atlas 计量,Karda 侧做归集视图);
- 以上入 Console 库详情与平台运维面;告警阈值 v1 定常量,v2 自适应。
