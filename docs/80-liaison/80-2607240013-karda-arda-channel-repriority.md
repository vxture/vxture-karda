## 2. karda 侧的方向调整（2026-07-23）

三条：

1. **karda 自建非结构化存储与向量库，自闭环支撑业务**——karda 持有内容的自有副本，
   不依赖任何外部源持续可达才能服务或重建自己的内容；
2. **陆续开放外接知识库/文档库能力**——SharePoint / Confluence / 对象存储 / 网盘等；
3. **Arda 视为内部三方**——一个"比较方便接的内部源"，与将来接的外部源**无架构地位差别**。

karda 已据此抽出通用层 `docs/30-design/220-connector-framework.md`，
`200-arda-channel` 自此为该框架的**第一个连接器实现**。

## 4. 通用层上提后，问题的问法也变了

抽出框架时发现一处值得同步：`200-arda-channel` 把 **notify-then-pull 当作默认传输方式**，
而那其实是 **Arda 的特例**——Arda 在同一平台内、有现成 C3 webhook 事件通道，所以能主动推通知；
一个外部文档库**通常没有任何方式主动通知 karda**。

框架因此把 **`poll` 定为默认能力、`notify` 为可选增强**，并把连接器差异建模为能力矩阵
（`change_detection` / `delivery` / `fetch` / `reconcile` / `delete_signal`）。Arda 在这个矩阵里的
取值是 `source / notify / ref / list / tombstone`——**是能力最完整的一档**，这也是它适合作为
框架首个验证实现的原因。

所以 §3 那五项换个问法就是：**请确认 Arda 在这五个能力维度上的实际取值**，其中
`reconcile=list`（一致性核对接口）是唯一尚不确定的一项。若 Arda 无法提供清单比对，
karda 会把 arda 连接器标注为"无对账能力"，长期运行的漂移只能靠全量重灌恢复——
这不阻塞接入，但影响它能承载什么内容（`220` §11 待拍板 #2）。
