## 1. A1 — embedding（karda 最硬的阻塞，KD-107）

karda 加工管线第四阶段 `embedding`：把 chunk 批量向量化，写入 karda 自有向量存储。

| # | 诉求 | 级别 | 理由 |
|---|---|---|---|
| A1.1 | 批量接口：单请求携带多个 chunk 文本，返回等长向量数组 | 【硬约束】 | 一份文档产生数十~数百 chunk；逐条往返吞吐不可接受 |
| A1.2 | **模型版本标识**随请求显式指定，且版本**可枚举、可固定** | 【硬约束】 | karda 是**库级锁定模型版本**——库创建时锁定，换版本 = 受控重建（`110-processing` §重建）。若 Atlas 只暴露"latest"、版本静默漂移，karda 无法保证同库向量同源，检索结果不可复现 |
| A1.3 | 响应含**向量维度**且同一模型版本维度稳定 | 【硬约束】 | 向量存储按维度建索引；维度漂移 = 索引失效 |
| A1.4 | service 模式鉴权，计量 workspace = **调用方传入的库归属 WS**（非触发用户） | 【硬约束】 | 加工成本记到资产归属方（`110-processing` §凭证姿态）。请确认 Atlas 计量能按请求携带的 workspace 归属，而非按调用方身份 |
| A1.5 | 429 区分**限流**与**配额耗尽**两种拒绝（见 §4 通用项） | 【硬约束】 | 见 §4 |
| A1.6 | 幂等或可重试：同一 chunk 文本 + 同版本重复请求返回一致向量 | 【期望】 | 重试与受控重建时避免向量抖动 |

**karda 侧参数**（供 Atlas 估容量，非要求 Atlas 适配）：单库文档量级不定；批量请求建议单批 ≤ 数百 chunk；
embedding 调用集中在 backfill 与重建期，可预期突发。

## 2. A2 — 解析类小模型（版面/OCR/表格/公式，KD-101）

karda deep-path 解析：扫描件/复杂版面 pdf/ppt → 版面分析 → 块类型识别 → 内容提取（OCR / 表格结构还原 /
公式转 LaTeX）→ 结构树。这些**视觉/小模型一律经 Atlas**（铁律，karda 解析器只含编排逻辑不内嵌运行时）。

| # | 诉求 | 级别 | 理由 |
|---|---|---|---|
| A2.1 | 解析类专用模型可**注册进 Atlas 模型目录**并经统一 runtime 调用 | 【硬约束】 | 否则 karda 被迫内嵌模型，违反铁律 |
| A2.2 | **批量接口**：单请求携带多页/多区域 | 【硬约束】 | 一份文档触发数十~数百次版面/表格调用；解析类是**高频、小请求、延迟敏感**，逐次往返会让加工吞吐被网络而非模型限制 |
| A2.3 | **部署亲和**：解析模型端点与 karda 加工 worker **同域部署**（低网络往返） | 【期望】 | 同 A2.2；若不可行 karda 需重估加工吞吐并调整三级队列并发池（`110-processing` §3），故请**明确告知可行性**，能力有无都要一个结论 |
| A2.4 | 请求/响应形态覆盖：版面框（bbox + 块类型）、OCR 文本（带位置锚点）、表格结构（行列/合并单元格）、公式（LaTeX） | 【期望】 | karda 解析输出统一为**元素树**（章节层级 + 元素类型 + 位置锚点），响应能直接喂元素树构建最优 |
| A2.5 | service 模式 + 按库归属 WS 计量（同 A1.4） | 【硬约束】 | 同 A1.4 |

**关于 A2.3**：这是 `70` 号函 §3 的核心诉求，也是 karda 无法自行绕过的一项。请把"能否同域部署 / 批量到
什么粒度"作为 Atlas 设计**解析类 runtime 时的一等约束**，而非事后优化——它直接决定 karda 加工域的
吞吐预算能否兑现。

## 3. A3 — rerank（cross-encoder，KD-102）

karda 检索统一精排：全命名空间候选并集 → 单次 rerank 出全局序（"召回廉价、精排一次"）。

| # | 诉求 | 级别 | 理由 |
|---|---|---|---|
| A3.1 | 单请求携带 (query, 候选文档数组)，返回每候选的**可全局比较分数** | 【硬约束】 | cross-encoder 对同一 query-doc 对打分天然全局可比——这正是 karda 不做跨索引分数归一的前提 |
| A3.2 | 候选池规模：**单次 ≤ 100 候选**（v1 最多 2 命名空间 × 50） | 【硬约束】 | 超过需分批，karda 侧已按 100 设计精排池上限 |
| A3.3 | **延迟预算：100 候选 P95 < 400ms**（在线检索路径，非离线） | 【硬约束 / 需 Atlas 确认可行性】 | 这是 `karda.search` 700ms 端到端里最大的一块。**若 Atlas 认为做不到，请尽早给出可承诺档位**——karda 会据此下调精排池 N（KD-009 联动），这是设计参数,越晚改代价越大 |
| A3.4 | 降级可判定：rerank 不可用时返回明确错误（非超时挂起） | 【硬约束】 | karda 降级契约 = rerank 失败回退 RRF 序并标注 `degraded`；需要一个**快速失败**信号，不能靠超时 |
| A3.5 | OBO 或 service 模式，计量归调用方 WS | 【硬约束】 | 检索由用户触发，计量归其 WS |

**A3.3 是本函里对 Atlas 排期最有价值的一条**：它是可行性问题不是实现细节，Atlas 越早给"100 候选能到
多少延迟"的判断，karda 的检索参数越早定稿。

## 4. 三项通用的语义诉求（Atlas 设计 runtime 时一并考虑）

| # | 诉求 | 级别 |
|---|---|---|
| G1 | **429 区分两种拒绝**：限流（可退避重试）vs 配额耗尽（应挂起）。两者不可返回无法区分的同一码 | 【硬约束】 |
| G2 | 计量按**请求携带的 workspace 归属**，不按调用方身份——加工/检索成本记资产或触发方 WS | 【硬约束】 |
| G3 | 模型 token 计量归 Atlas 上报，karda 只做归集视图，**互不重复计量** | 【硬约束】 |
| G4 | service 模式凭证：karda 后台加工以 service 身份铸币，`aud` / `act.sub` 语义对齐 product_210 | 【期望】 |

**G1 重申**：`90` 号回复说这项"尚无对应设计"——正因如此本函把它列为**硬约束**。若 Atlas runtime 把
限流与配额耗尽合并为同一个 429，karda 就无法正确选择"重试"还是"挂起为 `suspended_quota`"：配额耗尽
时会无谓重试打爆 Atlas，或限流时把本可成功的任务错误挂起。这是会在生产上出事的语义歧义，**请在
runtime 错误封套设计阶段就区分开**。

## 5. 优先级（供 Atlas 排期）

karda 的解锁顺序：**A1 > A3 > A2**。

- **A1（embedding）先到** → karda 加工管线向量化主路径脱离 parked、批次 5b 启动，检索域有可召回的东西；
- **A3（rerank）次之** → 批次 6b 统一精排落地；
- **A2（解析类）** → 提升 deep-path 解析质量，但 karda 可先用 fast-path（无视觉模型的文本类解析）产出可用管线，A2 是质量增强非启动前提。

即：**只要 A1 单项到位，karda 就能推进批次 5/6 的一大半**。若 Atlas 资源需排序，建议按此。
